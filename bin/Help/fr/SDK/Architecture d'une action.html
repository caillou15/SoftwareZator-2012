Architecture d'une action

L'architecture interne des actions pour SoftwareZator a entièrement changé pour cette nouvelle version. En revanche, pour les développeurs, elle ne change que partiellement.

Depuis la toute première version de SoftwareZator, celui-ci est un logiciel modulable, il fonctionne donc avec des plugins chargés au démarrage du logiciel. Un plugin permet de délivrer des actions pour SoftwareZator, qui pourront être utilisées dans l'éditeur de fonctions de cet outil.

Une action est définie par une classe hérité du type "VelerSoftware.Plugins3.Action" ou "VelerSoftware.Plugins3.ActionWithChildren".
Pour que cette action soit visible dans l'éditeur de fonctions (basé sur le Workflow 4.0 du Microsoft .Net Framework 4.0), il faut créer un Designer codé en XAML (donnant accès au WPF), basé sur le type "System.Workflow.ComponentModel.Design.ActivityDesigner".
Si votre action possède des paramètres modifiables par l'utilisateur, il sera alors conseillé de créer une fenêtre destiné à l'édition de ces paramètres. SoftwareZator fourni une fenêtre pré-construite délivrant une interface normalisé à tout les autres actions du logiciel. Il suffit d'hérité une fenêtre du type "VelerSoftware.Plugins3.ActionForm" pour utiliser cette fenêtre pré-construite.
SoftwareZator 2012 est disponible en Français et en Anglais. Il est important de traduire une action dans les deux langues afin qu'elle soit utilisable par un maximum d'utilisateur. Les textes tels que le nom de l'action, sa description et autre sont alors stockés dans trois fichiers ressources (fichiers *.resx) : un pour la langue neutre, un pour les textes en Français, et un pour les textes en Anglais.
